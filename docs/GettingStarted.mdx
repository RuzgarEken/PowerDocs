# Getting Started

Power flow consists of five elements: <Highlight>Actor</Highlight>, <Highlight>Stat</Highlight>, <Highlight>Effector</Highlight>, <Highlight>Effect</Highlight>, and <Highlight>Processor</Highlight>.

- <Highlight>Actor</Highlight>s contain <Highlight>Stat</Highlight>s.
- <Highlight>Effector</Highlight>s apply <Highlight >Effect</Highlight>s through <Highlight>Processor</Highlight>s to modify <Highlight>Stat</Highlight>s.

The overall concept is easy to understand.
In practice, using Power is mostly about defining relations between Power assets.

All actors store their stats in a multi layered container structure.

At the top level there is the Stat Repository. This holds the full stat definition set and is used by the Actor at runtime.

The stat hierarchy is structured as follows: <Highlight color="#ffffff">Repository -> Environment -> Container -> Stat</Highlight>

**Container**
  - Containers group Stat Definitions of the same type.
  - <img src={ContainerReference} alt="ContainerModel reference" />
  - For example
    - IntContainer contains Int stats
    - GeneratorFloatContainer contains GeneratorFloat stats
    - StateBasedEffectContainer contains StateBasedEffect stats

- **Environment** 
  - Environments group different types of stat containers together.
  - <img src={EnvironmentReference} alt="EnvironmentModel reference" />

- **Repository** 
  - The repository holds multiple stat environments.
  - <img src={RepositoryReference} alt="RepositoryModel reference" />
  - In simple systems, working only with the Default Environment may be sufficient.
  - However, you may also need systems that require multiple environments.
  - For example, imagine a robot with multiple limbs, each having its own health.
  - When we shoot the robot’s arm and its health reaches zero, we can disable that specific limb.
  - In this case, we can define separate environments for each limb such as the arm and other body parts.
  - This allows each environment to have different health values, damage multipliers, and unique reactions to damage events.

We use Effects to modify stats. Effects define the data describing how a stat should be changed. This Effect is passed through a process flow on the target, and the resulting value is applied to the stat as a modifier.
Let’s understand this process through an example.


Let’s say we want to decrease the enemy’s health by five.

For this, we need a Float Health stat in the StatRepository.
<img src={HealthStatReference} alt="HealthStat reference" />

We also need a <Highlight>FloatEffect</Highlight> with a value of minus five targeting the <Highlight>HealthStat</Highlight>.
<img src={EffectReference} alt="Damage Effect targeting the HealthStat" />

Effectors deliver their effects in Effect Packs, so we place our effect into <Highlight>Pack_Damage</Highlight>.
<img src={PackReference} alt="EffectPack which contains the damage effect" />

To process the effect, we use a "<Highlight>Processor_Add_Float</Highlight>" Processor assigned in the <Highlight>ProcessDefinition</Highlight>.
<img src={ProcessDefinition} alt="Process definition" />
<img src={AddProcess} alt="Float Add Processor" />

As you can see, we assign the effect’s ProcessorId as **Add_Float**, matching the processor in the <Highlight>ProcessDefinition</Highlight>.
The <Highlight>Damage</Highlight> effect is BaseStat(BS) effect without any attribute stats, so we use plain BSEffectProcess(For details about BaseStat and AttributeStat effects see the Effect Flow section)

After assigning the StatRepository and ProcessDefinition to the Actor, it looks like this: 
<img src={ActorEntity} alt="ActorEntity view with ProcessDefinition and StatRepository assigned" />

Finally, we use an On Collide Effector to apply the effect to the target.
<img src={OnCollideEffector} alt="OnCollideEffector view with Damage Pack assigned" />
